## Выбранная тема

**Тема 19 - Сервис генерации и валидации JWT токенов с ролями**

Система предназначена для управления аутентификацией в микросервисных приложениях.
Реализованы регистрация пользователей с ролями (admin, user, moderator),
выдача JWT access-токенов, их валидация и отзыв. Хранение данных - PostgreSQL.
Взаимодействие между сервисами - HTTP/REST.

## Практика №1 - Архитектурное проектирование в нотации C4

Разработаны три диаграммы в нотации C4 с использованием PlantUML:
- **C1 (Context)** - показывает взаимодействие системы с внешними акторами
- **C2 (Container)** - декомпозиция на API Gateway, Auth Service и PostgreSQL
- **C3 (Component)** - внутренняя структура Auth Service по паттерну Controller → Service → Repository

ИИ использовался для генерации кода PlantUML, после чего диаграммы дорабатывались вручную:
устранялись лишние компоненты, добавлялись протоколы взаимодействия и границы системы.

Подробный отчёт: [practice1/README.md](practice1/README.md)

## Практика №2 - Реализация MVP

Реализованы два микросервиса на Python FastAPI:

- **API Gateway** (порт 8000) - принимает внешние запросы, проксирует в Auth Service
- **Auth Service** (порт 8001) - бизнес-логика, работа с PostgreSQL, генерация JWT

Написаны 5 автотестов через pytest - все проходят успешно.
Локальный запуск через docker-compose с тремя контейнерами (gateway, auth, postgres).

ИИ сгенерировал ~90% кода. Вручную исправлена несовместимость версий
passlib и bcrypt (зафиксирована версия bcrypt==4.0.1).

Подробный отчёт: [practice2/PRACTICE2.md](practice2/PRACTICE2.md)

## Практика №3 - Деплой в Kubernetes (Minikube)

Все микросервисы развёрнуты в локальном кластере Minikube.
Созданы YAML-манифесты: Deployment, Service, Ingress, ConfigMap, Secret.

Конфигурационные параметры вынесены в ConfigMap и Secret.
Приложение доступно извне через Ingress по адресу `http://myapp.local`.
Успешно выполнены тестовые запросы через Ingress (HTTP 200).

Подробный отчёт: [practice3/PRACTICE3.md](practice3/PRACTICE3.md)

## Практика №4 - Мониторинг (Prometheus + Grafana)

Развёрнут стек мониторинга через Helm-чарт kube-prometheus-stack.
Auth Service инструментирован библиотекой prometheus-client - экспортирует метрики на `/metrics`.

Настроен ServiceMonitor для автоматического сбора метрик Prometheus.
Создан дашборд в Grafana с тремя панелями:
- RPS по эндпоинтам
- Среднее время ответа
- Количество выданных токенов (бизнес-метрика)

После нагрузочного теста (150 запросов) метрики отразили линейный рост токенов
и стабильную латентность ~0.01–0.02 с.

Подробный отчёт: [practice4/PRACTICE4.md](practice4/PRACTICE4.md)

## Заключение - рефлексия

### Как изменилось понимание разработки ПО после использования ИИ

До начала цикла казалось, что ИИ способен полностью автоматизировать разработку.
На практике выяснилось, что ИИ - это инструмент ускорения, а не замены разработчика.
Он хорошо генерирует шаблонный код, структуры проектов и конфигурационные файлы,
но не понимает контекст задачи глубоко: галлюцинирует лишние компоненты,
использует устаревшие версии библиотек, не учитывает специфику окружения.

Ключевой навык при работе с ИИ - умение проверять и критически оценивать результат,
а не слепо доверять сгенерированному коду.

### Где ИИ наиболее применим

- Генерация шаблонного кода (CRUD, конфигурации, Dockerfile, YAML-манифесты)
- Написание тестов по готовому коду
- Генерация документации и README
- Архитектурное прототипирование (первый вариант диаграмм)
- Поиск и объяснение ошибок в логах

### Где ИИ не справляется

- Отладка специфических конфликтов окружения (версии библиотек, платформо-зависимые баги)
- Проектирование нетривиальной бизнес-логики
- Принятие архитектурных решений с учётом долгосрочных последствий
- Работа с закрытыми или корпоративными системами без документации

### Как изменится рынок ИТ

ИИ-инструменты снизят порог входа в разработку и ускорят рутинные задачи,
что приведёт к перераспределению ролей: вырастет спрос на архитекторов,
DevOps-инженеров и специалистов по безопасности, способных проектировать
сложные системы и контролировать качество ИИ-генерированного кода.
Джуниор-разработчики, выполняющие только шаблонную работу, окажутся
под давлением автоматизации. Ценность специалиста будет определяться
умением работать в связке с ИИ, а не вопреки ему.
